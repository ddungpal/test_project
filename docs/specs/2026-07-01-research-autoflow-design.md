# 설계: 리서치 무중단 자동화 + 단일 최종검수 (research-autoflow)

_작성: 2026-07-01 · 상태: 설계 승인(구현 대기)_

## 문제

리서치 단계가 **후보 선택 → 검증 실행 → 검수 → 완료**의 다단계 위저드로 구현돼,
사용자(김짠부)가 5개 게이트를 직접 클릭/선택해야 한다. 이 중 진짜 '판단'은 사실상
**후보 선택(검증 범위=비용)** 하나뿐이고, 나머지는 결정이 없는 **기계적 통과 버튼**이다:

| # | 게이트 | 사용자 행위 | 진짜 판단? |
|---|---|---|---|
| 1 | 리서치 시작 (버튼) | 클릭 | ❌ |
| 2 | 후보 선택 (`ResearchScopeGate`) | 검증 항목 체크 | ✅ (비용·범위) |
| 3 | 리서치 검수 시작 (`EnterReviewButton`) | 클릭 | ❌ AI 0회·순수 상태전환 |
| 4 | 검수 승인 (`ResearchReview`) | 고위험 fact 승인/반려(기본=승인) | △ 고위험 있을 때만 |
| 5 | 다음 단계(대본) (버튼) | 클릭 | ❌ |

원칙은 **"김짠부는 매 단계 '선택'만"**인데, 구현이 **"모든 전이를 클릭"**으로 새어
"내가 다 운전하는" 피로감을 만들었다. → 사용하고 싶지 않게 만든다.

## 사용자 결정 (이 설계의 근거)

- **"결과만 보고 싶다."** 리서치는 크루가 알아서 끝까지 돌고, 사람 접점은 최소.
- **"예외는 스크립트 완성 후 한 번에 확인해도 늦지 않다."** 고위험 사실 검수를
  리서치 *중간*이 아니라 **스크립트 완성 *후* 한 화면에서** 처리.
- **검증 범위 기본값 = 정책 (나)**: 금융·고위험·핵심 숫자/비유만 자동 검증, 저위험
  주장은 출처만 표시. (= 사람이 어차피 검수하던 범위와 일치.)
- **시작 버튼 1개 유지**: 구성 확정 후 "리서치~스크립트 진행" 한 번 — 비용 발생 동의.

## 비전 (새 플로우)

주제·제목·썸네일·구성까지는 **지금처럼 김짠부가 선택**(진짜 창작 결정).
구성 확정 이후:

```
구성확정 →[한 번: "리서치~스크립트 진행"]→ (자동 스코프 → 자동 검증 → 자동 스크립트) → 스크립트 검수 1회 → 완료
              ↑ 유일한 시작 버튼(비용 동의)                                              ↑ 유일한 사람 접점
```

사람 접점: **5개 게이트 → 1개(최종 스크립트 검수)** + 시작 버튼 1개.

## 설계 상세

### A. 자동 스코프 (사람 선택 제거)

- `ResearchScopeGate`(사람 체크 UI)를 happy-path에서 제거.
- 검증 후보를 **정책 (나)로 자동 선택**: **기존 '검수 에스컬레이션' 술어를 그대로
  검증 선택 기준으로 재사용**한다 — 즉 *검증하는 것 = 어차피 사람이 확인했을 것*
  (금융/고위험 + 숫자/비유 필요 항목). 저위험 주장은 검증 안 하고 출처만.
- **비용 하드캡 적용**: 자동 선택분이 캡을 넘으면 우선순위(고위험 우선)로 잘라
  운영 $10 안에 든다. 잘린 항목은 최종 검수에서 "미검증·출처만" 으로 표시.
- 신규 정책/술어 발명 금지 — `researchScope`/escalation 로직 재사용.

### B. 무중단 전이 (중간 게이트 제거)

- `research_ready`에서 **멈추지 않는다.** "검수 시작" 버튼·`research_review` 사람
  게이트를 happy-path에서 우회.
- 에스컬레이션된(고위험) 사실은 검수로 빼지 않고 **스크립트까지 운반** — 최종
  검수에서 인라인으로 노출.
- `research_scoped → researching → research_ready → (검증완료)`는 Inngest 내부에서
  자동 진행(사람 입력 없음).

### C. 자동 체인 (리서치 → 스크립트)

- 검증 완료 시 `run/script.requested`를 **자동 발행**(지금은 사람이 누름).
- 새 Inngest 이벤트/함수 추가 없이 기존 라우팅 재사용(스테이지 체이닝).

### D. 단일 최종 검수 (`ScriptReview` 확장)

- 완성된 스크립트 본문 + 검증/고위험 사실을 **인라인 칩**으로 표시(출처 포함,
  "확인 필요" 표식, 미검증은 별도 표식).
- **전체 승인 = 완료**(기본값, 한 번 클릭).
- **개별 사실 반려 → 그 사실을 쓴 세그먼트만 재생성**(`researchReconcile` 부분
  재생성 경로 재사용). 반려는 드물 것으로 가정 — 재작업 비용 < UX 이득.

## 안 깨지는 것 (불변식)

- **거버넌스 "팩트체크 = 사람 최종확인" 유지** — 제거가 아니라 **위치만 끝으로** 이동.
  (출처명시 + 교차검증 + 사람 최종확인 3요소 모두 보존.)
- `could_not_verify`(날조 금지)는 차단하지 않고 최종 검수에서 플래그.
- **비용 하드캡 $10** 유지 — 정책(나) + 캡 컷.
- 창작 선택 단계(주제·제목·썸네일·구성)는 **변경 없음**.
- 진행 중 표시(`progress_note`/스테퍼)는 유지 — 자동 진행 동안 "깜깜이" 방지.

## 작업 범위 (대략 harness phase 2~3개)

| 영역 | 변경 |
|---|---|
| pipeline | 상태전이 자동체인(`researchStage`→script), `research_review` 게이트 우회, 검증완료→`run/script.requested` 자동 발행 |
| scope | 자동선택 = 에스컬레이션 술어 재사용(`researchScope`/`researchGate`), 비용캡 컷 |
| UI 제거 | `ResearchScopeGate`·`EnterReviewButton`·중간 단계 버튼(happy-path) |
| UI 확장 | `ScriptReview`에 인라인 사실 칩 + 반려→세그먼트 재생성 |
| reconcile | 최종 반려 시 부분 재생성(`researchReconcile`) |

## 미해결 / 구현 시 결정

- 재진입(`ResearchReentryActions`: 다시 선택/다시 검증/예시 재생성)은 **고급 옵션으로
  잔존** 가능 — happy-path 게이트는 아니되, 최종 검수에서 "리서치 다시" 진입점 제공 여부는
  구현 단계에서 결정.
- 인라인 칩 ↔ 스크립트 세그먼트 매핑 키(어떤 fact가 어떤 segment에 쓰였는지) — 기존
  `script_segments`/`explanation_asset`/fact 연결 스키마 확인 후 확정.
