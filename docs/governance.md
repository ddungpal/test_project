## 2. 댓글 작성자 개인정보 — **최소수집 (확정)**

- **작성자 닉네임·채널ID 등 식별정보는 보관하지 않는다.** ingest 시 `author` 컬럼 자체를 두지 않음(스키마 차원 보장 — tech.md `comments_raw`).
- **유튜브 댓글ID(`external_id`)는 HMAC 해시**(`external_id_hash`)로만 저장 — dedup·삭제동기화는 가능하되 작성자 역추적은 차단.
- **댓글 본문만** 보관(주제 발굴·약신호 분석에 본문이면 충분).
- 근거: 작성자 식별정보를 보관할 업무상 이유가 없음 → 최소수집 원칙.

## 3. 보관기간 & 삭제 — **확정**

- **원본 보관**: 내부 학습 도구이므로 원본(댓글 본문·스크립트·리서치 출처)은 기본 **무기한 보관**.
- **삭제 요청**: 시청자 등 삭제 요청이 오면 **해당 댓글을 파기**한다. `comments_raw`는 L1 불변성 트리거 **예외 테이블**로 두어 삭제가 가능하게 한다(service-role 전용 경로: 행 DELETE 또는 `body=null`+`redacted_at` 레닥션). 그 외 L1 원본은 불변 유지.
- **소스 동기화 정리**: YouTube에서 삭제된 댓글은 다음 동기화 때 우리 DB에서도 정리(`external_id_hash` 매칭).
- **저장량 알림**: DB가 과도하게 커져 정리가 필요한 시점이 오면 **대시보드 알림으로 통지**(자동 삭제 X, 사람이 판단). → tech.md 텔레메트리에 저장량 모니터 추가.

## 4. 외부 API로의 데이터 전송 — **확정 (C안)**

리서치·생성 과정에서 데이터가 외부 LLM/검색 API와 DB로 전송된다. **무엇이 나가는지**를 명시하고, **댓글 원문은 LLM에 통째로 넘기지 않는다**(C안).

### 4.1 전송 데이터 표 (무엇이 어디로)
| 데이터 | → 목적지 | 전송 여부 | 비고 |
|---|---|---|---|
| 스크립트 원문·리서치 자료 | Anthropic/OpenAI API | ✅ 전송 | 생성·검증에 필요. API는 학습 미사용·단기 보관. |
| 리서치 쿼리 | Tavily·Perplexity·공식도메인 | ✅ 전송 | 검색어만. |
| **댓글 원문** | LLM API | ❌ **전송 안 함** | 아래 4.2 — 집계·키워드만 LLM에. |
| YouTube 토큰·API 키·service role | 어디에도 | ❌ **프롬프트 금지** | 서버 전용 env. |
| 전체 작업물 | Supabase(전용 DB) | ✅ 저장 | 우리 소유 DB. |

### 4.2 댓글 처리 규칙 (C안 — 원문 비전송)
- 댓글 **원문(시청자 사연·재무상황 등)은 LLM 프롬프트에 넣지 않는다.**
- 댓글→주제 발굴은 **집계·약신호·키워드 추출 결과**만 LLM에 전달한다. (예: "ISA 관련 질문 급증", "빚 상환 vs 투자 고민 다수" 같은 **신호**면 충분 — 개별 사연 원문 불필요.)
- 키워드/집계 추출 단계는 **로컬 처리 우선**(규칙 기반·임베딩 클러스터링). 원문이 외부로 나가야 하는 경우엔 민감 수치·개인사 **레닥션 후** 전송.
- 근거: 시청자 개인 사연 원문이 국외 LLM 서버까지 갈 업무상 이유가 없음(주제 발굴엔 신호로 충분). principles.md §5 혼동 방지·최소노출과 일관.

### 4.3 선결 규칙
- **민감 자격증명(YouTube 토큰·API 키·service role)은 절대 LLM 프롬프트에 넣지 않는다.** 서버 전용 env.
- 외부 LLM/검색은 **API 경로만** 사용(소비자 키 혼용 금지) — 기본 학습 미사용·단기 보관 보장.
